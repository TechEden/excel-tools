# Excel 批处理工具-excel-tools

一个本地运行的 Excel `.xlsx` 批处理工具，面向日常台账处理中的高频重复工作：把总表按部门、班组、责任人等字段拆成多个文件，也可以把文件夹里一批 Excel 合并成总表。

项目提供网页界面，用户可以上传文件或复用本地缓存，确认 Sheet、列名所在行、拆分字段、拆分值、输出列、公式保留策略、合并排序等设置，然后生成结果文件。核心目标是减少手工筛选、复制、粘贴、另存为，并让处理过程可见、可停止、可复用。

> 开发和维护约定见 [PROJECT_REQUIREMENTS.md](PROJECT_REQUIREMENTS.md)，里面记录了长期需求、增强功能边界、配置阶段轻量化、日志、UI 偏好和输出命名规则。

## 项目背景

本项目的开发初衷来自国有企业推进国产化办公过程中遇到的真实场景。部分单位虽然仍可使用 Windows 系统，但受数据安全、内外网隔离、软件准入和终端管控影响，很多在线工具、闭源小工具或仅支持特定 Windows 环境的开源软件无法直接使用。业务人员在处理台账、清单、报表时，仍然需要频繁依赖 Excel/WPS 手工筛选、复制、粘贴、另存为，既耗时，也容易出错。

这个工具尝试解决的是“办公效率”和“安全合规”之间的矛盾：

- 数据不上传到第三方平台，所有 Excel 处理都在本机完成。
- 后端基于 Python、Flask、openpyxl，便于在 Windows、macOS、Linux 和国产化桌面环境中部署。
- 使用浏览器作为界面入口，降低业务人员使用门槛。
- 针对台账拆分、批量合并、公式处理、表头识别、配置复用等实际办公痛点做专项优化。
- 运行结果、错误提示、进度和预计剩余时间尽量可见，减少“程序卡住了还是还在处理”的不确定感。

项目并不追求替代专业 BI 或 ETL 平台，而是聚焦“日常 Excel 批处理”这一类高频、刚需、可标准化的办公流程，让用户在安全受控的本地环境中完成快捷处理。

## 适用场景

- 按人员、客户、部门、地区等字段拆分 Excel 明细表
- 从一个大表中批量生成多个子文件
- 一个工作簿包含多个 Sheet，需要按同一个字段统一拆分
- 不按分类字段，只按固定行数把一个或多个 Sheet 切成多个文件
- 文件夹中有多个 Excel，需要按 Sheet 分开合并成结果总表
- 拆分时只导出部分字段，隐藏不需要的列
- 控制公式是否保留，避免结果文件继续依赖原公式
- 同一类台账不同版本反复处理，希望保存拆分配置下次复用
- 文件不希望上传到第三方平台，需要本地处理

## 核心功能

### 文件载入

- 支持上传 `.xlsx` 文件
- 支持本地缓存复用，避免重复上传同一个文件
- 支持缓存文件版本选择
- 支持查看当前文件的基础信息和字段概览

### 按分类单 Sheet 拆分

- 选择一个 Sheet 进行拆分
- 支持自动识别表头为第 1 行到第几行，也可以手动调整
- 支持选择按哪一列的分类进行拆分
- 支持选择要拆分的字段值
- 支持选择不输出列，默认输出全部列
- 支持选择需要保留公式的列；启用保留公式或按列保留公式时属于增强功能
- 支持自定义文件后缀
- 支持自定义输出目录
- 支持选择拆分路径分配方式：
  - 按文件归档：`output/原文件名/Sheet_时间/Sheet名_分类值_时间.xlsx`，适合按一次拆分任务集中回看
  - 按分类归档：默认保存为 `output/原文件名/分类列名/分类值/Sheet名_分类值_时间.xlsx`，例如 `output/weather_multi_sheet_test/记录人/A/Sheet名_A_时间.xlsx`；如果手动指定输出目录，则保存为 `输出目录/分类列名/分类值/Sheet名_分类值_时间.xlsx`
  - 分类归档目录名默认使用用户选择的拆分列名，也可手动改成 `记录人分类`、`城市分类`、`乡镇归档` 等更贴合业务的名称
- 支持拆分进度显示
- 支持停止正在执行的拆分任务

### 按分类多 Sheet 拆分

适用于一个 Excel 工作簿中多个 Sheet 结构相近，并且希望按同一个字段统一拆分的情况。

例如一个工作簿包含 `春 / 夏 / 秋 / 冬` 四个 Sheet，每个 Sheet 都有 `记录人` 字段。工具可以按 `记录人` 一次性拆分，生成：

- `A` 的 Excel，里面包含 A 在各个 Sheet 中的数据
- `B` 的 Excel，里面包含 B 在各个 Sheet 中的数据
- `C` 的 Excel，里面包含 C 在各个 Sheet 中的数据
- `D` 的 Excel，里面包含 D 在各个 Sheet 中的数据

如果某个拆分值在某个 Sheet 中不存在，该 Sheet 会自动跳过，不会生成空 Sheet。

多 Sheet 拆分支持：

- 选择参与拆分的多个 Sheet
- 每个 Sheet 自动识别列名所在行（表头范围），并可单独调整
- 只显示所选 Sheet 都有的共同字段，降低选错风险
- 不输出列按每个 Sheet 单独选择，默认输出各 Sheet 的全部列
- 基础拆分默认写入公式缓存值，不保留公式
- 可按每个 Sheet 单独选择需要保留公式的列；此能力属于增强功能
- 可配置字段别名，例如把不同 Sheet 中的“归属部门”“责任部门”统一成同一个拆分字段；此能力属于增强功能
- 可选择保留未参与拆分的其他 Sheet；此能力属于增强功能
- 输出文件名先显示参与输出的 Sheet 组合，再显示分类值；多个 Sheet 名之间用 `+` 连接，例如 `夏+秋_A_260707_22时58分.xlsx`
- 支持按文件归档或按分类归档：
  - 按文件归档：`output/原文件名/Sheet组合_拆分结果_时间/Sheet组合_分类值_时间.xlsx`
  - 按分类归档：默认保存为 `output/原文件名/分类列名/分类值/Sheet组合_分类值_时间.xlsx`，例如 `output/weather_multi_sheet_test/记录人/A/夏+秋_A_时间.xlsx`；如果手动指定输出目录，则保存为 `输出目录/分类列名/分类值/Sheet组合_分类值_时间.xlsx`

### 固定行数拆分

适用于不按部门、人员等分类值拆分，只想把大表按固定数据行数切成多个 Excel 的场景。

例如一个 Sheet 有 2 行表头和 520 行数据，设置每个文件 200 行数据后，会生成：

- 第 1 段：保留表头 + 数据第 1-200 行
- 第 2 段：保留表头 + 数据第 201-400 行
- 第 3 段：保留表头 + 数据第 401-520 行

固定行数拆分支持：

- 选择一个或多个 Sheet
- 每个 Sheet 自动识别列名所在行（表头范围），并可单独调整
- 每个结果文件都保留对应 Sheet 的表头
- 自定义每个文件包含多少行数据，例如 100 行或 200 行
- 选择拆分后要输出的列，默认全选
- 选择需要保留公式的列，默认不保留公式；启用保留公式或按列保留公式时属于增强功能
- 多 Sheet 时同一段号写入同一个结果文件
- 输出文件夹包含“每份多少行”信息
- 输出文件名包含数据行范围，例如 `第001段_数据第1-100行_台账完整度_260705_21时08分.xlsx`

### 多表合并

适用于文件夹里有一批结构相近的 Excel，需要汇总成总表的场景。

多表合并支持：

- 选择本地文件夹或 ZIP 压缩包，也支持手动输入路径
- 扫描 `.xlsx` 文件并忽略临时锁文件
- 先加载文件夹文件清单，再加载所选文件的 Sheet，避免一次性读取过慢
- 支持按文件名关键字筛选，并提示是否按筛选结果作为合并清单
- 默认全选扫描到的文件，可手动全选、清空或勾选
- Sheet 合并项默认不勾选，由用户手动选择要合并的 Sheet
- 自动识别列名所在行（表头范围），保留第一份表头，后续数据跳过重复表头
- 支持手动修改表头起止行，例如第 2 行到第 3 行
- 合并结果按 Sheet 分开输出，不把不同 Sheet 混到一个 Sheet 里
- 支持对最终合并结果按指定列升序或降序排序
- 默认写入公式缓存值；选择保留公式时属于增强功能
- 合并进度包含读取、合并、排序、保存阶段
- 支持预计剩余时间和停止合并
- 输出文件默认保存到 `output/来源名称/来源名称_多表合并_时间.xlsx`
- 对 openpyxl 无法正常读取样式的文件，会尝试保留数据并使用降级样式输出

### 配置复用

很多台账不是一次性处理，而是同一套规则反复用于不同月份、不同版本的文件。工具支持把拆分设置保存为配置文件，减少重复操作。

配置复用支持：

- 开始拆分时弹出现代页面内弹窗，询问是否保存当前配置
- 可选择“保存并开始”、“仅本次拆分”或“取消”
- 配置名称可重命名，不填时按日期自动命名
- 配置文件保存在 `configs/split/`
- 文件名使用用户可读的配置名称，例如 `部门月度台账拆分配置.json`
- 重名时自动追加 `_2`、`_3`
- 下次上传或选择缓存文件后，可载入最近配置或手动选择配置文件
- 按分类多 Sheet 配置载入后，会自动加载共同字段和拆分值，并提示未匹配内容
- 配置载入后不会自动开始拆分，最终仍由用户确认

### 数据概览

工具提供类似 `pandas.info()` 或 KNIME 数据概览的字段分布视图，帮助拆分前快速了解当前 Sheet 的数据情况。

概览内容包括：

- Sheet 名称
- 数据行数
- 列数
- 空单元格比例
- 字段类型识别
- 字段样例值
- 文本字段：非空、空值、唯一值
- 数值字段：最小值、最大值、平均值、中位数

### 公式处理

工具支持两种公式处理方式：

- 默认不保留公式，导出 Excel 中已缓存的计算结果
- 可选择保留全部公式，或只保留指定公式列

注意：openpyxl 不能主动计算公式。如果原 Excel 文件没有被 Excel/WPS 保存过计算结果，公式缓存值可能为空。

## 普通功能与增强功能设计

工具的功能分级原则是：**基础拆分、合并和常规输出作为普通功能开放；明显增加处理复杂度、耗时或节省大量人工核对的能力作为增强功能。**

普通功能包括：

- 按分类单 Sheet 拆分
- 按分类多 Sheet 拆分
- 固定行数拆分
- 多表合并
- 上传、本地缓存复用、预览、字段统计
- 选择 Sheet、拆分字段、拆分值和不输出列
- 自定义文件后缀、输出目录和基础归档方式
- 保存和载入拆分配置

增强功能建议包括：

- 保留公式：默认输出公式计算值；启用保留公式或按列保留公式时，作为增强功能处理。
- 保留其他 Sheet：拆分结果中额外带入未参与拆分的 Sheet，会增加文件体积和处理耗时，作为增强功能处理。
- 多 Sheet 字段别名配置：当不同 Sheet 的列名不一致时，将多个列名统一成同一个业务字段，作为增强功能处理。
- 高级配置复用：包含字段别名、保留公式、保留其他 Sheet 等增强项的配置，可作为后续增强能力继续细化。

当前页面采用“基础功能 + 增强项可选”的展示方式：用户可以直接进行单 Sheet 拆分、多 Sheet 拆分、固定行数拆分和多表合并；只有勾选或配置增强项时，页面才会提示本次将使用增强功能，并在开始前弹窗确认消耗 1 次本月 VIP 额度。未启用增强项时，这些基础流程不消耗增强次数。

## 运行日志

工具会把关键操作写入本地运行日志，便于排查“上传后长时间等待”“读取 Sheet 失败”“拆分或合并中断”等问题。

- 页面入口：在“本地缓存”区域点击 `下载运行日志`。
- 本地路径：源码模式默认在 `logs/app.log`；便携模式默认在压缩包内的 `data/logs/app.log`。
- 日志内容：接口状态码和耗时、上传保存与解析、缓存读取、Sheet 数据读取、拆分/合并任务开始、完成、取消和异常堆栈。
- 日志轮转：单个日志文件最大约 5MB，最多保留 5 个历史文件。

## 启动方式

项目支持两种启动方式，代码只维护一份：

- 本机 Python 启动：适合开发者，或电脑上已经安装 Python 的用户。
- 绿色便携启动：适合无法安装 Python、无法使用 `.exe` 的国产化办公环境；便携包应通过 GitHub Releases 发布。

### 方式一：使用本机 Python 启动

适用条件：

- 已安装 Python 3.9 或更高版本。
- 可以使用 `pip` 安装依赖。
- 可以打开本机浏览器访问 `127.0.0.1`。

Linux / 国产化桌面环境 / macOS 可执行：

```bash
./scripts/start_local.sh
```

macOS 也可以双击：

```text
scripts/start_local.command
```

Windows 可双击：

```text
scripts/start_local.bat
```

脚本会自动完成：

1. 在项目目录创建 `.venv` 虚拟环境。
2. 安装 `requirements.txt` 中的依赖。
3. 设置本地运行参数。
4. 启动 Flask 服务。

启动后访问：

```text
http://127.0.0.1:5001
```

如需手动启动，也可以执行：

```bash
python -m venv .venv
. .venv/bin/activate
pip install -r requirements.txt
FLASK_HOST=127.0.0.1 FLASK_PORT=5001 FLASK_DEBUG=0 python app.py
```

### 方式二：绿色便携启动

绿色便携版用于“用户不安装 Python，也不运行 `.exe`”的场景。它的形式是一个压缩包，内置 Python 运行环境、项目代码和依赖库。

便携包建议通过 GitHub Releases 发布，不建议把 `runtime/python` 直接提交到源码仓库。

便携包解压后结构类似：

```text
excel-tool-portable-uos-x64/
├── runtime/
│   └── python/             # 内置 Python 运行环境
├── app/                    # 项目代码
├── data/
│   ├── uploads/            # 上传缓存
│   ├── output/             # 输出结果
│   └── configs/            # 用户配置
├── start.sh                # Linux / 国产化桌面启动
├── start.command           # macOS 启动
├── start.bat               # Windows 启动
└── README_PORTABLE.md
```

用户使用流程：

1. 在 GitHub Releases 下载与本机系统和 CPU 架构匹配的绿色包。
2. 解压压缩包。
3. 运行 `start.sh`、`start.command` 或 `start.bat`。
4. 浏览器访问 `http://127.0.0.1:5001`。

便携版启动后，上传文件、输出结果和配置文件默认保存在压缩包内的 `data/` 目录，便于整体复制、备份和迁移。

### 构建绿色便携包

构建便携包需要在目标系统或兼容系统上准备一份可独立运行的 Python runtime。不同系统和 CPU 架构需要分别构建，例如：

- `excel-tool-portable-uos-x64.zip`
- `excel-tool-portable-kylin-x64.zip`
- `excel-tool-portable-linux-arm64.zip`
- `excel-tool-portable-macos-arm64.zip`

构建命令示例：

```bash
PYTHON_RUNTIME_DIR=/path/to/standalone/python \
PLATFORM_TAG=uos-x64 \
./scripts/build_portable.sh
```

构建脚本会生成：

```text
dist/excel-tool-portable-uos-x64/
```

确认可运行后，可压缩并上传到 GitHub Releases：

```bash
cd dist
zip -r excel-tool-portable-uos-x64.zip excel-tool-portable-uos-x64
```

关键参数说明：

- `PYTHON_RUNTIME_DIR`：必填，指向目标系统可独立运行的 Python 目录。
- `PLATFORM_TAG`：可选，用于区分发布包平台，例如 `uos-x64`、`kylin-x64`、`linux-arm64`。
- `APP_DATA_DIR`：运行时数据目录。源码模式默认是项目根目录；便携模式默认是压缩包内的 `data/`。
- `FLASK_HOST`：默认 `127.0.0.1`，不建议改成公网地址。
- `FLASK_PORT`：默认 `5001`。
- `FLASK_DEBUG`：默认 `0`，发布时保持关闭。

## 代码结构

项目已按后续扩展方向拆分为轻量模块：

- `routes/`：按功能拆分接口，包括首页、配置、拆分、合并和多 Sheet 拆分。
- `services/`：放置任务状态、进度和业务服务出口，后续可继续承接更完整的业务流程。
- `excel_utils/`：放置 Excel 读取、样式兼容、文件名清洗、Sheet 名处理等工具函数出口。
- `config_store/`：放置本地配置读写、配置摘要和配置路径管理出口。
- `core.py`：保留公共依赖、应用实例和当前复用较多的 Excel 处理函数，便于在不破坏现有功能的前提下逐步迁移。

这次拆分以“保持现有功能稳定”为优先目标，接口地址保持不变。后续继续扩展时，可以逐步把 `core.py` 中的函数迁移到 `services/`、`excel_utils/` 和 `config_store/` 内部实现中。

## 文件格式说明

当前仅支持 `.xlsx` 文件。

不支持 `.xls`，因为项目使用 openpyxl 处理 Excel 文件，而 openpyxl 不支持旧版 `.xls` 格式。

## 项目定位

这个工具不是复杂的数据分析平台，而是一个专注于 Excel 日常批处理流程的本地办公辅助工具。它的目标是让常见的手工拆表、合表工作变得可配置、可预览、可停止、可重复执行，尽量减少用户在 Excel 中反复筛选、复制、粘贴、另存为的操作。

从开源项目角度看，它更适合作为一个可落地的本地办公自动化样例：功能来自真实业务场景，技术栈保持轻量，逻辑尽量透明，便于二次开发和单位内网部署。

## 适合谁使用

- 经常处理 Excel 明细表的运营、财务、行政、人事、销售支持人员
- 需要按人员、部门、客户、地区拆分表格的业务人员
- 需要把多份 Excel 台账按 Sheet 分别汇总的业务人员
- 希望在本地完成 Excel 批处理、不想把文件上传到在线平台的用户
- 需要一个简单可改造的 Flask + openpyxl 示例项目的开发者

## 注意事项

- 建议先用小文件验证拆分规则，再处理大文件
- 保留其他 Sheet 会增加处理时间，大文件场景建议关闭
- 如果拆分值超过 30 个，工具会提示确认，避免误选导致生成大量文件
- 公式缓存值依赖 Excel/WPS 预先保存的计算结果
- 配置文件只保存处理规则，不保存原始 Excel 内容
- 载入旧配置后仍建议检查 Sheet、字段、拆分值和输出列是否匹配当前文件
- 本工具面向可信本地环境使用，不建议直接暴露到公网

## 安全与合规设计

- 本地处理：上传文件、缓存文件、配置文件和输出结果都保存在项目本地目录或用户指定目录中。
- CORS 限制：默认仅允许 `127.0.0.1` 和 `localhost` 对应端口访问接口。
- 下载限制：下载接口只允许访问项目输出目录或本次运行已登记的输出目录，避免任意路径下载。
- 文件校验：上传和合并扫描均校验 `.xlsx` 是否为有效 ZIP 结构，并忽略 Excel 临时锁文件。
- 文件名处理：输出文件名、Sheet 名和上传文件名会进行非法字符清洗，降低路径和文件名异常风险。
- ZIP 处理：读取 ZIP 时逐个筛选 `.xlsx` 文件，忽略 `__MACOSX`、临时文件和非 Excel 内容，不使用整包无差别解压。
- 敏感数据控制：`.gitignore` 默认排除上传缓存、输出结果、配置 JSON、IDE 文件和系统文件，避免误提交业务数据。

## 来自实际工作的功能沉淀

这些功能不是为了堆功能而做，而是来自真实台账处理过程中的重复痛点：

- 表头不一定都在第 1 行，所以所有拆分和合并流程都支持自动识别表头，并允许手动调整为“列名所在行”。
- 业务表经常有多个 Sheet，拆分和合并都不能简单地把所有 Sheet 混成一个结果，需要保留 Sheet 结构。
- 有些结果文件不应继续保留公式，默认输出缓存值；确实需要公式时，再按列选择保留。
- 大文件处理时间不可预测，所以拆分和合并都需要进度、阶段说明、预计剩余时间和停止按钮。
- 同一类文件会反复处理，所以需要配置复用，避免每次重复选择 Sheet、字段、输出列和公式列。
- 固定行数拆分解决的是“按批次切大表”的需求，和按分类拆分不同，不能要求用户选择拆分值。
- 多表合并必须先列文件，再读所选文件的 Sheet，否则大量文件场景会显得很慢。
- 文件名筛选只改变展示列表，不应悄悄改变用户已勾选的合并清单，因此提供“使用当前筛选结果”的显式操作。
- 合并排序作用于最终合并结果，而不是每个源文件内部，避免用户对排序结果产生误解。
- 某些国产化或 WPS/Excel 混合环境生成的 `.xlsx` 可能存在样式兼容问题，工具会尝试去除异常样式读取数据，并用 `宋体 9号` 作为降级输出样式。
- 合并 ZIP 来源可以减少“先解压、再选择文件夹”的手工步骤，更贴近日常收到压缩包后的处理习惯。

## 关键调试与优化记录

- 修复多 Sheet 合并时全部内容被写入同一个 Sheet 的问题，改为按 Sheet 分别合并并保留各自表头。
- 将多表合并流程拆成“加载来源文件清单”和“加载所选文件的 Sheet”，避免大量文件场景一开始就读取所有工作簿。
- 修复搜索筛选后误把筛选结果当作合并清单的问题，确保合并依据用户手动勾选结果。
- 增加合并/拆分后台任务、进度轮询、取消任务和预计剩余时间展示。
- 修复公式未选择保留时仍写出公式的问题，默认严格导出缓存值。
- 增加固定行数拆分，并在输出目录和文件名中记录“每份 X 行”和“数据第 X-Y 行”。
- 增加拆分配置保存、读取、按文件匹配、按类型排序和现代页面弹窗，减少重复配置成本。
- 优化多 Sheet 别名配置，改为默认手动输入，必要时再加载列名，并支持加载中提示和取消。
- 处理 openpyxl 读取异常样式导致的 `Fill() takes no arguments`、`expected openpyxl.styles.fills.Fill`、`list index out of range` 等问题。
- 增加从 `.xlsx` 压缩包 XML 中读取 Sheet 名的兜底逻辑，降低样式损坏导致 Sheet 列表为空的概率。
- 对下载接口、CORS、上传文件名和运行默认参数做安全加固，降低开源发布后的误用风险。
